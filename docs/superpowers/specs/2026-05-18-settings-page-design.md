# Settings Page Design

**Date:** 2026-05-18
**Branch:** ai-factory
**Reference:** `./aif/ui/ai-factory/` (AI Factory reference implementation)

## Goal

Add a working Settings page to the SUSE AI Lifecycle Manager. This page lets platform engineers configure GitOps (Fleet), SUSE Application Collection credentials, SUSE Registry credentials, and advanced air-gap options. It mirrors the settings page in `./aif/ui/ai-factory/` across all four layers: CRD, Go backend, Helm chart, and Vue UI extension.

## Settings CRD Schema

Group: `ai-platform.suse.com`, version `v1alpha1`, kind `Settings`. Namespaced scope. The singleton CR is named `"settings"` and lives in the operator's release namespace.

### `templates/settings.yaml`

```yaml
apiVersion: ai-platform.suse.com/v1alpha1
kind: Settings
metadata:
  name: settings
  namespace: {{ .Release.Namespace }}
  labels:
    {{- include "suse-ai-operator.labels" . | nindent 4 }}
  annotations:
    helm.sh/resource-policy: keep
spec: {}
```

The `helm.sh/resource-policy: keep` annotation means user-configured settings survive chart upgrades and uninstalls. `spec: {}` is intentionally empty вҖ” users populate it through the UI.

### `templates/api-service.yaml`

Exposes port 8080 via a ClusterIP Service so the Rancher proxy URL
`/k8s/clusters/local/api/v1/namespaces/<ns>/services/http:suse-ai-operator:8080/proxy`
resolves to the operator's HTTP API.

## Go Backend (`suse-ai-operator/`)

### New files

**`api/v1alpha1/settings_types.go`**

CRD Go types with full kubebuilder marker annotations. Uses `corev1.SecretKeySelector` for all secret references (standard Kubernetes type, no custom wrapper needed). Kubebuilder markers produce the same validation constraints as the AIF CRD.

After editing: run `make generate` (deepcopy) and `make manifests` (CRD YAML).

**`internal/api/handler.go`**

```go
type APIHandler struct {
    Client    client.Client
    Namespace string  // operator release namespace; Settings CR lives here
}

func (h *APIHandler) RegisterRoutes(mux *http.ServeMux)
```

`RegisterRoutes` is the single registration point for all `/api/v1/*` routes. Future endpoints (apps, workloads, etc.) are added here by adding handlers and wiring them in this method вҖ” no changes to `cmd/main.go` required.

**`internal/api/settings.go`**

- `GET /api/v1/settings` вҖ” fetches `settings` CR from `h.Namespace`. Returns 404 + `{ error: "NOT_FOUND", message: "..." }` when the CR doesn't exist (UI shows "not configured yet" info banner, matching AIF behaviour).
- `PUT /api/v1/settings` вҖ” accepts `{ spec: {...} }` JSON body. Applies via Server-Side Apply with field manager `"suse-ai-operator-api"`. Returns the full updated CR. SSA creates-or-updates atomically, so no prior GET is needed.

Both handlers share `writeJSON(w, v)` and `writeError(w, code, err)` helpers that produce consistent JSON envelopes.

**`internal/controller/settings/settings_controller.go`**

```go
type SettingsReconciler struct {
    client.Client
    Scheme *runtime.Scheme
}
```

Reconcile stamps `status.lastApplied = now` and patches the status subresource. This is the hook where future stories apply settings to engines (Fleet client, registry client, etc.). RBAC markers: `get;list;watch;create;update;patch;delete` on `settings` and `get;update;patch` on `settings/status`.

### Updated files

**`api/v1alpha1/groupversion_info.go`** вҖ” no change needed; Settings registers itself via `init()` in `settings_types.go`.

**`cmd/main.go`**

1. Add `--api-bind-address` flag (default `:8080`)
2. Add `--operator-namespace` flag (default: read from `NAMESPACE` env var, fallback `"suse-ai-operator"`)
3. Register `Settings` in the scheme
4. Wire `SettingsReconciler` into the manager
5. Start HTTP server in a goroutine before `mgr.Start`:

```go
mux := http.NewServeMux()
apiHandler := &api.APIHandler{Client: mgr.GetClient(), Namespace: operatorNamespace}
apiHandler.RegisterRoutes(mux)
srv := &http.Server{Addr: apiBindAddr, Handler: mux}
go func() {
    <-ctx.Done()
    srv.Shutdown(context.Background())
}()
go srv.ListenAndServe()
```

## UI Extension (`pkg/suse-ai-lifecycle-manager/`)

### New files

**`utils/operator-api.ts`**

Mirrors AIF's `operator-api.ts`. Builds the base URL from `MANAGEMENT_CLUSTER`, `OPERATOR_NAMESPACE`, `OPERATOR_SERVICE`, `OPERATOR_PORT` constants. Exports:

```typescript
export function getSettings(): Promise<any>
export function putSettings(spec: any): Promise<any>
```

`OperatorError` extends `Error` with `.status: number` and `.code: string` so the Settings page can branch on `e.status === 404`.

**`pages/Settings.vue`**

Direct port of `aif/ui/ai-factory/pkg/ai-factory/pages/settings.vue`. Two adaptations only:
1. Import paths: `'../utils/operator-api'` and `'../config/types'` (adjusted for lifecycle manager directory layout)
2. i18n key prefix: `aif.pages.settings.*` вҶ’ `suseai.pages.settings.*`

All four accordion sections are preserved as-is: Fleet/GitOps, SUSE Application Collection, SUSE Registry, Advanced.

### Updated files

**`config/types.ts`** вҖ” add operator service coordinates:

```typescript
export const MANAGEMENT_CLUSTER = 'local';
export const OPERATOR_NAMESPACE = 'suse-ai-operator';
export const OPERATOR_SERVICE   = 'suse-ai-operator';
export const OPERATOR_PORT      = 8080;
```

**`config/suseai.ts`** вҖ” add Settings entry to `VIRTUAL_TYPES` array so the nav sidebar shows a Settings link.

**`routing.ts`** вҖ” replace the placeholder `component: () => import('./pages/Apps.vue')` on the Settings route with `component: () => import('./pages/Settings.vue')`.

**`l10n/en-us.json`** вҖ” add all settings strings under `suseai.pages.settings` (title, apply, notConfigured, all section/field labels). Translated from AIF's `en-us.yaml` to JSON format.

## Data Flow: Save Settings

```
User edits form вҶ’ clicks Apply
  вҶ’ Settings.vue: save() вҶ’ buildCrdSpec(this.spec)
  вҶ’ operator-api.ts: putSettings(crdSpec)
  вҶ’ PUT /api/v1/settings { spec: crdSpec }
  вҶ’ Rancher proxy вҶ’ suse-ai-operator:8080
  вҶ’ settings.go: handlePutSettings
  вҶ’ Server-Side Apply вҶ’ Kubernetes API
  вҶ’ Settings CR updated
  вҶ’ SettingsReconciler: Reconcile() stamps status.lastApplied
  вҶ’ HTTP response: updated CR JSON
  вҶ’ Settings.vue: this.spec = buildSpec(data.spec)  вҶҗ UI reflects saved state
```

## Data Flow: Load Settings (404 case)

```
Settings.vue: fetch()
  вҶ’ getSettings() вҶ’ GET /api/v1/settings
  вҶ’ settings.go: CR not found вҶ’ 404 { error: "NOT_FOUND" }
  вҶ’ OperatorError.status === 404
  вҶ’ this.notFound = true, this.loaded = true
  вҶ’ UI renders info banner: "No settings configured yet. Fill in the form and save."
  вҶ’ Empty form shown, PUT will create the CR via SSA
```

## Out of Scope

- Settings validation webhook (not implemented in AIF either at this stage)
- Applying settings to engines (Fleet client reconfiguration, registry client update) вҖ” controller stub is the hook; wired in future stories
- Settings for fields beyond what the AIF UI page covers (AIFactorySubscription, Entitlements, Nvidia) вҖ” not surfaced in the UI page
- E2E tests вҖ” unit test for `handleGetSettings` / `handlePutSettings` is in scope
